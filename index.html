daVinci-LLM 的 Data Mixture 方案详解

2.1 数据分类体系: Data Darwinism Framework (L0-L9)

daVinci-LLM采用Data Darwinism框架 [Qin et al., 2026, arXiv:2602.07824] 对所有数据源按处理深度进行L0-L9分级标注:

  • L0 Data Acquisition: 从网页、PDF、代码平台等收集原始数据
  • L1 Format Normalization: 统一格式转换(OCR、HTML解析等)
  • L2 Rule-based Filtering: 基于规则的过滤(去重、短文过滤、语言过滤)
  • L3 Lightweight Model Filtering: 轻量模型过滤(教育价值评分、领域分类)
  • L4 Generative Refinement: 生成式精炼(用LLM去噪、结构重组，保持语义等价)
  • L5 Cognitive Completion: 认知补全(用LLM重建推理链、术语解释、教学桥接)
  • L6-L9 更高阶的合成(上下文补全、环境合成、生态合成、世界合成)
关键洞察: 处理深度(processing depth)是与数据量(volume)同等重要的优化维度。在固定数据量的条件下，从L2提升到L3-L5可以系统性地提升模型能力。

2.4 关键消融实验发现

(1) 数据处理深度的影响
  • L2→L3 代码过滤: MBPP +3.40, Overall +0.27
  • Baseline →L4 数学精炼: MATH +7.00, GSM-8K +1.37
  • L5 Synthetic QA: 展现强领域特异性(CodeQA提升编程，CC-QA提升科学)
(2) 训练动态
  • General knowledge: 在 ~1T tokens 内快速饱和
  • Code和Science: 在 4T+ tokens 仍持续增长
  • 引入QA > 仅调整比例: Science +20.34 vs +15.75, Code +2.62 vs +2.15
(3) 数据混合设计
  • Code-Science平衡至关重要: C-30-S-30 (均衡) 优于 C-40-S-20 (偏向代码)
  • QA浓度的阶段依赖性: Stage 2-1中Code在30%后崩溃; Stage 2-2中30%/50%/70%均单调提升
结论: 渐进式混合策略——先保守建立基础(30% QA)，再激进增强(70% QA)
(4) 评估方法的影响
  • PPL评估和Generative评估会导致不同的模型排名
  • 建议同时使用两种评估协议

关键原则: 预训练阶段仅包含大规模自然数据源(不含task data/QA)。Task data保留到midtraining阶段——因其规模不足以在预训练尺度产生影响，且会干扰消融实验。

3.3 Constrained Data Mixing (Olmix) 方法

(A) Base Procedure — 基于swarm
  1. Swarm Construction: 训练大量30M参数代理模型(3B tokens), Dirichlet采样混合比例, swarm大小 = 5 × 领域数
  2. Per-task Regression: 每个代理模型提供 (混合权重 → 任务BPB) 数据点, 拟合GLM
  3. Mix Optimization: 最小化平均任务BPB, 约束重复不超4-7x
(B) Conditional Mixing — 元流程
  • 将已优化的混合作为"虚拟领域"冻结，仅在新领域上重新优化
  • Round 1: 优化24个WebOrganizer类别
  • Round 2: 优化编程语言混合 (固定web:code=75:25)
  • Round 3: 整合PDF数据

3.4 Quality-Aware Upsampling

  • 对每个WebOrganizer的24个主题分别生成上采样曲线
  • 底部40%数据丢弃，顶部5%数据重复7次
  • 相比flat filtering，在数据受限场景下表现更好
  • 效果: 平均BPB改善0.056，54个任务中仅13个退化(均<0.035)

3.6 Midtraining 方法论

Microanneal — 分布式轻量测试
  • 5B目标数据 + 5B web数据 = 10B混合退火
  • 与10B web-only基线对比
  • 快速、廉价评估单个数据源的影响
Integration Test — 集中评估
  • 定期汇总microanneal结果
  • 100B候选混合完整退火训练
  • SFT训练后评估后训练效果
  • 共进行5轮集成测试

3.7 Midtraining 关键发现

  • 领域间显著trade-off: Math-code-thinking提升Math/Code但损害MC-QA/GenQA
  • Thinking/Instruction有利: 包含vs不包含 — Average: 50.7 vs 48.8 (+1.9)
  • 特殊token应留在SFT: midtraining中加入chat template导致GSM8K从49.43降至0

4.6 关键共识

  1. 预训练后期应增加推理密集型数据(code, math, science)的比例
  2. 结构化QA数据对推理能力提升至关重要，且效果优于仅调整领域比例
  3. 领域间存在显著trade-off，需要在目标能力和通用能力之间平衡
  4. 不同能力维度饱和速率不同: 通用知识快速饱和，推理能力持续提升
  5. 数据质量(处理深度/重写)比简单扩大数据量更有效
  6. 渐进式课程学习优于静态混合——数据配方应随训练进度调整

Data Mixture 相关文献综述

5.1 混合优化方法

1RegMix [Liu et al., 2024] — arXiv:2407.01492

将数据混合优化建模为回归问题。训练大量小型代理模型(不同混合比例)，拟合回归模型预测任意混合的性能。OLMo 3 Olmix直接采用swarm-based approach。

2Data Mixing Laws [Ye et al., 2025] — arXiv:2403.16952

拟合参数化缩放定律来建模混合比例与性能的关系。提供显式函数形式，可解析或数值优化混合方案。

3CLIMB [Diao et al., 2025] — arXiv:2504.13161

基于聚类的迭代引导方法优化数据混合。与RegMix的一次性方法互补。

4DoReMi [Xie et al., 2024] — NeurIPS 2024

使用分布鲁棒优化(DRO)进行领域重加权。280M代理模型可使8B模型训练加速2.6x。

5Olmix [Chen et al., 2026]

Conditional mixing——将已优化混合作为"虚拟领域"冻结，仅在新/变化领域上重新优化。大幅降低重新优化成本。

5.2 数据处理与质量

6Data Darwinism [Qin et al., 2026] — arXiv:2602.07824 | 7DCLM | 8WebOrganizer [Wettig et al., 2025] — 24个主题分类

9Nemotron-CC [Su et al., 2024] — arXiv:2412.02595 | 10FineMath | 11MegaMath [Zhou et al., 2025] | 12olmOCR [Poznanski et al., 2025]

5.3 LLM重写/数据合成

13SwallowMath/Code — CraneMath microanneal: MATH +18.5, GSM8K +27.4

拟定 Data Mixture 实验方案

6.1 混合优化方法选择

方案A: Swarm-based (OLMo 3路线)
训练~100个30M代理模型, Dirichlet采样, 拟合GLM搜索最优混合
方案B: 消融驱动 (daVinci路线)
~1B模型, 500B tokens对照消融, 每次变化一个维度
方案C: Microanneal (推荐先行)
从已有checkpoint出发, 每个候选数据源5-10B microanneal, 与web-only基线对比
建议: 先用方案C快速筛选数据源，再用方案A或B确定最终比例。

补充分析: MiMo-7B, MiMo-V2-Flash, DeepSeek-V4 NEW

7.1 MiMo-7B

Xiaomi7B dense~25T tokens

数据来源与处理

  • 数据来源: 网页、学术论文、书籍、编程代码、合成数据
  • 开发专门HTML提取工具，优化数学公式和代码块保留 (trafilatura常丢失)
  • 增强PDF解析处理STEM和代码内容
  • URL去重 + MinHash全局去重
  • 微调小型LLM作为数据质量标注器(替代启发式规则过滤器，后者会错误移除数学/代码网页)
关键发现: 合成推理数据可以训练极多epoch而不会过拟合(与非推理数据截然不同)。

三阶段混合策略

Stage 1: 所有数据源(除合成推理响应)。下采样广告/新闻/招聘等低价值内容，上采样专业领域高质量数据。上下文: 8,192 tokens
Stage 2: 数学+代码增加到 ~70%。增强专业技能而不损害通用能力。上下文: 8,192 tokens
Stage 3: 加入 ~10% 合成推理响应(数学/代码/创意写作)。上下文扩展 8K → 32K

(7) 合成/LLM重写数据在后期阶段大量使用

daVinci(L4/L5) | OLMo 3(CraneMath/CraneCode) | MiMo(合成推理响应) | DeepSeek-V4(领域专家蒸馏)

(8) 专业化数据提取工具的重要性

MiMo-7B明确指出: 通用提取器(trafilatura)常丢失推理密集内容(数学公式、代码块)。标准工具对推理内容存在系统性缺陷。

(9) 学习率调度与数据混合阶段对齐

所有论文都将学习率变化点与数据分布变化点对齐。引入专业化数据时通常降低学习率或进入cosine decay。
